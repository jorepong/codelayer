# [U1] 커밋 후 발송 흐름 조립 · 01 커밋된 것만 보낸다

여기서부터는 개념 설명이 아니에요. **발송 흐름이 만들어지는 과정을 옆에서 지켜보는** 자리예요. 완성된 코드를 논리 순서로 풀어 주는 게 아니라, 실제로 짜 나간 *시간·의존 순서*를 따라가며 "이 순간 무엇을 보고, 무엇을 결정했나"를 함께 볼 거예요. 당신이 능숙한 개발자의 작업 화면을 곁에서 관전한다고 생각하세요. 중간중간 제가 **"당신이라면 여기서 뭘 할 것 같아요?"**를 물을 텐데, 잠깐 멈춰 스스로 답해보고 다음 문단으로 확인하세요 — 보기만 하면 "나도 하겠다"는 착각이 생기거든요.

> **먼저 정직하게.** 이 관전이 다루는 건 **7단계 시점의 배선**이에요. 지금 워킹 트리의 코드는 8·9단계에서 이 배선이 **릴레이 + Kafka 워커로 갈려 나가** 이 형태가 아니에요(그래서 `dispatchPending` 같은 메서드는 지금 파일에 없어요). 하지만 여기서 내릴 **핵심 판단 — "커밋된 것만 발송한다"**— 은 그대로 8단계 Outbox의 씨앗이 되니, 사라진 게 아니라 자란 거예요. 코드는 journal(구현 일지)을 척추로 그때 모양으로 복원해 보여드릴게요.

## 출발점 — 무엇이 이미 있고, 무엇을 이어야 하나

단계 6까지 와서, `수집 → 매칭 → Notification 생성(PENDING)`이 **동기로 한 트랜잭션 안에서** 돌고 있어요. 아직 메일은 안 나가요 — 알림이 DB에 `PENDING`으로 앉아만 있죠. 이번 [U1]의 일은 딱 하나예요. **그 `PENDING`을 실제 메일 발송으로 잇고, 성공하면 `SENT`로 넘기는 것.**

그런데 이 "잇기"가 생각보다 자리를 잘 골라야 하는 일이에요. 아무 데나 `mailSender.send()`를 부르면 될 것 같지만, **어디서 부르냐**에 이번 단계의 모든 판단이 걸려 있어요. 짜 나간 순서를 따라가 볼게요.

## 비트 1 — 발송 로직을 어디에 둘까

가장 먼저 마주친 갈림길이에요. 발송 로직을 **어느 클래스에 넣을 것인가.**

> 🔮 **예측해보세요.** 단계 6의 생성은 `NotificationService.createForPosting()`이 맡고 있어요. 발송도 알림에 관한 일이니, 여기에 `send…` 메서드를 하나 더 붙이는 게 자연스러워 보이죠. 당신이라면 발송을 이 생성 서비스에 넣을까요, 아니면 따로 뺄까요? 뺀다면 왜요?

작업자는 **생성 서비스에 넣지 않고, `NotificationDispatchService`를 새로 갈라** 뒀어요. 왜일까요? 지금 당장의 편의로는 한 서비스에 몰아넣는 게 짧은데도요.

여기서 관전의 핵심이 나와요 — **앞을 봤기 때문이에요.** 작업자는 8단계를 알고 있었어요. 8단계에서 이 발송을 **Kafka 발송 워커가 다시 불러야** 하는데, 생성(`createForPosting`)과 발송이 한 서비스에 엉켜 있으면 그때 "발송만" 떼어내기가 어려워져요. 그래서 *지금* 값을 치르더라도 발송을 독립된 서비스로 분리해 둔 거예요. 이건 "지금 필요해서"가 아니라 "곧 필요할 걸 알아서" 미리 그은 경계예요.

> **관전 포인트 — 이게 사람의 판단이에요.** journal에 이 결정이 "이유: 단계 8에서 이 발송을 Kafka 발송 워커가 재사용해야 한다. 미리 분리해 둔다"로 적혀 있어요. 능숙한 개발자는 눈앞의 한 줄이 아니라 *다음 단계에서 이 코드가 어떻게 불릴지*를 보고 경계를 그어요. 당신 프로필에도 "진입점을 얇게 두어 로직을 분리하면 입력 경로가 달라도 재사용된다"는 걸 스스로 도출했다고 적혀 있죠 — 그 원리가 여기서 "미리 분리"라는 행동으로 나온 거예요.

## 비트 2 — 발송기를 인터페이스로 (여기는 빠르게 지나갑니다)

그다음 발송기를 `NotificationMailSender`(인터페이스) + `SmtpNotificationMailSender`(구현)로 세웠어요. **이건 [2]에서 이미 깊이 봤으니 빠르게 지나갈게요** — repository가 DB 세부를 감추듯 발송기가 SMTP를 `infra` 뒤에 감추는 그 일반화고, 인터페이스로 둔 진짜 값이 테스트 대역이었다는 것도 [2]에서 다뤘죠. 여기서 결정할 게 없어요. 정형 배선이라 눈만 스치고 넘어갑니다.

## 비트 3 — 핵심: 발송을 트랜잭션 *밖*에서 부른다

이번 관전의 정수예요. 발송을 *어디서 트리거하느냐*.

지금 `수집 → 저장 → 매칭 → 생성`이 `JobPostingCollectionService.collect()` 안에서 **하나의 `@Transactional`로** 묶여 돌아요. 발송을 여기에 잇는 가장 짧은 길은, `collect()` 맨 끝에 발송 한 줄을 더하는 거예요. 그 트랜잭션 안에서요.

> 🔮 **예측해보세요.** 발송을 `collect()`의 트랜잭션 *안*에 두면, 무엇이 깨질까요? 힌트 — 메일은 한번 나가면 되돌릴 수 없고, 트랜잭션은 롤백될 수 있어요. 이 둘이 만나면?

작업자는 발송을 트랜잭션 **안에 두지 않았어요.** 대신 워커에서 이렇게 짰어요 — journal이 기록한 7단계 당시 모양으로 복원하면:

`JobCollectionWorker.java` (7단계 시점 — 지금은 8단계에서 이 호출이 걷혔어요)
```java
public void run() {
    collectionService.collect();        // ❶ @Transactional — 저장·생성, 여기서 커밋된다
    dispatchService.dispatchPending();  // ❷ 커밋 후, 트랜잭션 밖에서 발송
}
```

`❶`의 `collect()`가 자기 트랜잭션을 열고 닫아요(커밋). 그리고 **커밋이 끝난 뒤에** `❷`가 별도로 발송을 불러요. 왜 이렇게까지 갈라놨을까요?

메일은 **비가역 부수효과**예요. 만약 발송을 `collect()`의 트랜잭션 *안*에서 했다고 해봐요. 발송을 한 뒤, 같은 트랜잭션의 뒷부분에서 무언가 실패해 **롤백**되면 어떻게 될까요? DB는 트랜잭션을 통째로 되감아 "아무 일도 없던" 상태로 돌아가요 — 그런데 **메일은 이미 날아가 버렸어요.** 되돌릴 수가 없죠. 그럼 "존재하지 않는 알림"의 메일이 사용자에게 가 있는, DB와 현실이 어긋난 상태가 돼요.

그래서 규칙은 이거예요 — **"확정(커밋)된 것만 발송한다."** 발송을 트랜잭션 경계 *밖*으로, 커밋이 끝난 뒤로 밀어내면, 롤백된 알림은 애초에 발송 대상이 되질 않아요(커밋을 못 했으니). 이 한 줄의 배선 — `❶` 다음에 `❷`, 사이에 커밋 — 이 그 규칙을 코드로 만든 거예요.

> **관전 포인트 — 이 감각이 8단계로 자란다.** 지금은 "커밋 후에 발송한다"를 워커의 순차 호출로 소박하게 풀었어요. 8단계에서 이게 **Outbox 패턴**이 돼요 — "커밋된 `PENDING`만 릴레이가 폴링해 발행한다"로요. 원리는 똑같아요, 규모만 커진 거예요. 당신이 지금 이 트랜잭션 경계 감각을 잡아두면, 8단계 Outbox가 "새로운 것"이 아니라 "이걸 비동기로 늘린 것"으로 보여요.

아래 그림이 이 타이밍을 한눈에 보여줘요 — 왼쪽은 발송을 트랜잭션 *안*에 뒀을 때 롤백이 부르는 사고, 오른쪽은 커밋 후로 밀어낸 안전한 배선이에요.

![두 배선의 대비. 왼쪽: 발송을 트랜잭션 안에 두면, 발송(메일 나감) 뒤 롤백이 일어나도 DB만 되감기고 메일은 회수 불가 — DB엔 없는 알림의 메일이 현실에 남는다. 오른쪽: collect가 커밋으로 트랜잭션을 닫은 뒤 발송하면, 롤백된 알림은 애초에 발송 대상이 아니라 이 사고가 원천 차단된다.](assets/transaction-boundary-dispatch.svg)

## 비트 4 — 가지 않은 길들 (작업자가 여기서 본 갈림길)

능숙한 개발자는 한 길을 고를 때 *다른 길들을 봤다가 안 간* 거예요. journal에 그 갈림길이 남아 있어요. 사람의 박자로 보여드릴게요.

**갈림길 A — 스프링 이벤트(`@TransactionalEventListener(AFTER_COMMIT)`).** 스프링엔 "트랜잭션이 커밋된 *후에* 이벤트 리스너를 실행"하는 기능이 있어요. "커밋 후 발송"이 정확히 이거라, 여기 딱 맞아 보이죠. 작업자도 이 길을 봤어요. 그런데 **안 갔어요** — 8단계에서 어차피 발송을 릴레이 폴링 방식으로 바꿀 거라, 지금 이벤트 리스너를 도입하면 8단계에서 **다시 걷어내야** 하거든요. 곧 버릴 걸 지금 들이는 건 낭비죠. 그래서 "워커에서 순차 호출"이라는 더 소박한 길을 택했어요. (이게 능숙함이에요 — 더 세련된 도구가 있어도, 곧 갈아엎을 자리엔 안 써요.)

**갈림길 B — 발송을 알림마다 개별 트랜잭션으로 쪼개기.** 발송 하나하나를 독립 트랜잭션으로 격리하는 방법도 있어요(한 건 실패가 다른 건에 안 번지게). 그런데 이것도 **안 갔어요** — 7단계는 동기 단순 버전이라 과해요. 재전송·중복 격리는 8·9단계의 몫이라, 지금 끌어오면 단계의 경계가 흐려져요. "지금 필요한 만큼만"이죠.

> **관전 포인트 — 멈춤과 빨리감기.** 보세요, 비트 2(인터페이스)는 눈만 스치고 지나갔는데 비트 3~4(트랜잭션 경계·버린 대안)는 천천히 팠어요. *어디서 깊이 보고 어디서 빨리 지나가나* 그 자체가 수업이에요 — 능숙한 개발자는 모든 줄에 똑같은 힘을 쓰지 않아요. 정형 배선은 흘리고, 판단이 걸린 자리에만 멈춰요. 완벽주의로 모든 줄을 똑같이 파고들면 정작 중요한 데 쓸 힘이 없어요.

## 비트 5 — dispatchPending 안쪽 (가볍게)

마지막으로 발송 메서드 자체예요. journal 기준 7단계 모양은 이래요.

```java
@Transactional
public void dispatchPending() {
    for (Notification n : notificationRepository
            .findByStatusOrderByCreatedAt(PENDING)) {   // ❸ PENDING 조회, fetch join
        try {
            mailSender.send(n.getUser().getEmail(), subject(n), body(n));
            n.markSent();                                // ❹ 성공 → SENT
        } catch (RuntimeException e) {
            n.markFailed();                              // ❺ 실패 → FAILED, 나머지는 계속
        }
    }
}
```

`❸`은 `PENDING` 알림을 오래된 순으로 가져오는데, **fetch join**으로 `user`·`jobPosting`을 함께 끌어와요(발송에 email·title·url이 필요한데 LAZY면 N+1이 나니까 — 이건 당신이 JPA에서 이미 아는 거라 가볍게 짚고 넘어가요). `❹`·`❺`가 상태 기계 전이예요. `@Transactional`이라 `markSent()`가 필드만 바꿔도 **dirty checking**으로 저장돼요(`save()` 명시 호출 없이 — 이것도 당신이 아는 거죠). `❺`처럼 한 건이 실패해도 `try/catch`로 격리해 나머지 발송은 계속돼요.

이 메서드 안쪽은 대부분 당신이 이미 자율적으로 다루는 것들이라(상태 전이·fetch join·dirty checking) 관전에선 가볍게 지나가요. 이 유닛의 진짜 값은 *메서드 안쪽*이 아니라 *메서드를 어디서 부르느냐*(비트 3)에 있었으니까요.

> **지금 코드와 비교 — 정직하게.** 워킹 트리에선 이 `dispatchPending`이 8·9단계에서 **둘로 갈렸어요** — 발행을 맡는 `relayPending()`과 한 건 발송을 맡는 `sendOne()`으로요. `findByStatusOrderByCreatedAt`(❸)은 그대로 살아남아 지금은 릴레이가 써요. 즉 7단계의 이 "한 메서드가 순회 발송"이, 8단계에서 "릴레이가 발행 → 워커가 건별 발송"으로 펼쳐진 거예요. 궁금하면 지금 `NotificationDispatchService`를 열어 `relayPending`·`sendOne`을 보세요 — 이 관전의 `dispatchPending`이 어떻게 갈라졌는지 눈으로 확인돼요(단, 거기 붙은 Kafka·Redis는 8·9 얘기라 지금은 눈길만).

## 관전은 여기까지 — 이제 당신 차례

흐름을 다 봤어요. 이번 [U1]의 뼈대는 딱 하나로 줄어요 — **"생성은 트랜잭션 안, 발송은 커밋 후 트랜잭션 밖."** 나머지(인터페이스·상태 전이·fetch join)는 당신이 이미 아는 조각들이 배선된 것뿐이에요.

관전이 길어지면 "조금만 더 보고 시작하자"는 미루기가 돼요. 충분히 봤으니 여기서 멈추고, **직접 해보기**로 넘어가요. 관전은 *짜는 것*의 준비였을 뿐이에요 — 다음 단계(장악 평가)에서 이 배선을 당신 손으로 재구성하거나, 일부러 심어둔 결함을 잡아내면서 "읽어서 안다"를 "직접 짜고 잡아낸다"로 바꿀 거예요.
