# [U1] 커밋 후 발송 흐름 조립 — 장악 평가 (결함 주입)

> **상황.** 어떤 개발자(혹은 AI)가 단계 7 "메일 발송"을 아래처럼 짰습니다. **컴파일도 되고, 평소 정상 흐름에서는 통합테스트도 초록불**이에요. 발송도 잘 나가고, `SENT` 전이도 잘 돼요. 그런데 이 배선에는 [U1]에서 본 **원칙을 어기는 결함이 하나** 숨어 있습니다. 리뷰어가 되어 잡아내 보세요.
>
> **이게 왜 좋은 훈련인가.** AI가 짠 코드는 대개 이래요 — 컴파일되고 happy-path는 통과하는데, 경계 조건에서 조용히 깨져요. "그럴듯하지만 취약한 코드"를 읽고 결함을 짚어내는 건, AI 시대에 점점 더 중요해지는 실전 능력이에요.

## 리뷰 대상 코드 (결함이 심겨 있음)

`JobPostingCollectionService.java`
```java
@Service
public class JobPostingCollectionService {

    private final JobFeedSource feedSource;
    private final JobPostingRepository jobPostingRepository;
    private final NotificationService notificationService;
    private final NotificationDispatchService dispatchService;

    // 생성자 생략 (네 협력자 모두 주입)

    @Transactional
    public CollectionResult collect() {
        List<JobPosting> fetched = feedSource.fetch();
        int saved = 0;
        for (JobPosting posting : fetched) {
            if (jobPostingRepository.existsBySourceUrl(posting.getSourceUrl())) {
                continue;                                  // 이미 수집된 공고 — 멱등
            }
            JobPosting persisted = jobPostingRepository.save(posting);
            notificationService.createForPosting(persisted);   // 매칭 → Notification(PENDING) 생성
            saved++;
        }
        dispatchService.dispatchPending();                 // 생성된 PENDING을 메일로 발송
        return new CollectionResult(fetched.size(), saved);
    }
}
```

`NotificationDispatchService.java`
```java
@Service
public class NotificationDispatchService {
    ...
    @Transactional
    public void dispatchPending() {
        for (Notification n : notificationRepository
                .findByStatusOrderByCreatedAt(NotificationStatus.PENDING)) {
            try {
                mailSender.send(n.getUser().getEmail(), subject(n), body(n));
                n.markSent();
            } catch (RuntimeException e) {
                n.markFailed();
            }
        }
    }
}
```

## Q2 — 왜 문제인가

그게 왜 결함인지 설명하세요. **어떤 상황**에서 **무엇이** 깨지나요?
