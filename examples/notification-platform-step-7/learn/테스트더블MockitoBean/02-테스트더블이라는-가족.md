# [2] 테스트더블 / `@MockitoBean` · 02 테스트더블이라는 가족

〈01 왜 가짜가 필요한가〉에서 "진짜 객체 자리에 끼우는 가짜 대역"이 테스트더블이라고 했죠. 그런데 "가짜"라고 뭉뚱그리면 이번 단계의 핵심을 놓쳐요. 가짜에도 **성격이 다른 종류들**이 있고, 우리 코드엔 그중 **둘**이 실제로 들어와 있거든요. 그 둘의 차이가 이번 [2] 학습의 알맹이라, 먼저 가족 전체를 지도로 펼쳐 놓고 시작할게요.

## 왜 이름이 '더블'인가

`test double`의 '더블'은 영화의 **스턴트 더블(stunt double)**에서 온 말이에요. 위험한 장면에서 진짜 주연 배우 대신 뛰어드는 대역이죠. 관객(테스트를 부르는 코드)이 보기엔 그 장면에 배우가 있는 것 같지만, 실제로 그 자리에 선 건 대역입니다. 테스트더블도 똑같아요 — 코드 입장에선 `NotificationMailSender`가 있는 것 같지만, 실제로 그 자리에 선 건 진짜가 아닌 대역이에요.

대역은 "무엇을 대신해 주느냐"에 따라 성격이 갈립니다. 흔히 다섯으로 나눠요. 아래를 한 줄씩 보되, **끝의 두 개(fake·mock)에 눈을 두세요** — 우리 코드에 실제로 있는 게 그 둘이니까.

## 다섯 종류의 대역

**Dummy(더미).** 자리만 채우는 껍데기예요. 넘겨야 하니까 넘기긴 하는데, 그 테스트에서 **실제로 쓰이진 않는** 인자. 예를 들어 생성자가 로거를 요구하는데 이 테스트에선 로그를 안 볼 거라면, 아무것도 안 하는 로거를 던져 주는 식이죠. "있어야 해서 있을 뿐, 아무 역할도 안 함"이 더미예요.

**Stub(스텁).** **정해진 답을 돌려주는** 대역이에요. "이 메서드를 부르면 항상 이 값을 리턴해"라고 미리 심어 둡니다. 진짜 대신 예측 가능한 응답을 주는 게 목적이에요. 예: 환율 조회 대신 "무조건 1300원"을 돌려주는 대역. 스텁은 **테스트에 필요한 입력 상황을 만들어 주는** 쪽입니다.

**Spy(스파이).** 진짜(또는 진짜에 가깝게) 동작하면서, 동시에 **자기가 어떻게 불렸는지 몰래 기록**하는 대역이에요. 스텁이 "정해진 답 주기"라면, 스파이는 "진짜로 일하되 호출 내역을 남기기"에 방점이 있어요.

**Fake(페이크).** **진짜처럼 실제로 동작하지만, 운영에는 못 쓸 만큼 가벼운/단순한 구현**이에요. 대표 예가 인메모리 DB예요 — 진짜 DB처럼 저장·조회가 되지만, 프로세스가 끝나면 사라지는 가벼운 버전이죠. 사실 당신은 이미 fake를 매 통합테스트에서 쓰고 있었어요. `@SpringBootTest`가 띄우는 그 인메모리 DB가 운영 DB의 fake거든요. 진짜 MySQL 대신, 진짜처럼 굴지만 가벼운 대역이 그 자리에 서 있던 거예요.

**Mock(목).** **자기가 어떻게 불렸는지를 기록해 두고, 나중에 "이렇게 불렸어야 한다"를 검증하게 해 주는** 대역이에요. 여기서 무게 중심이 완전히 바뀝니다. 지금까지 넷(dummy·stub·spy·fake)은 "테스트가 굴러가게 **상황을 만들어 주는**" 쪽이었어요. mock은 반대예요 — **그 대역을 향한 호출 자체가 검증의 대상**이 됩니다. "메일 발송기가 정확히 이 주소로, 딱 한 번 불렸는가?"를 mock에게 나중에 캐물을 수 있어요.

> **한눈에 가르는 기준.** 넷은 *들어오는 것*(내가 부르면 무엇을 돌려주는가)을 다루고, mock은 *나가는 것*(내가 그 대역을 어떻게 불렀는가)을 다뤄요. 발송처럼 "돌려받을 값은 없고(`void`), 대신 '제대로 불렀는지'가 전부인" 상호작용을 검증할 땐 mock이 딱 맞습니다. 발송의 `send()`가 `void`였던 걸 떠올리면, 왜 여기서 mock이 주인공인지 자연스럽죠.

## 우리 코드에 실제로 있는 둘

이 프로젝트의 발송 테스트 언저리에는 이 다섯 중 **둘**이 들어와 있어요.

**하나 — `TestNotificationMailSender` (fake에 해당).**
```java
@Component
@Profile("test")
public class TestNotificationMailSender implements NotificationMailSender {  // ❶
    @Override
    public void send(String to, String subject, String body) {
        // no-op                                                            // ❷
    }
}
```
`❶`에서 발송기 계약을 진짜로 구현하지만, `❷`에서 보듯 **아무것도 안 해요.** 진짜 발송기가 설 자리를 대신 채워 주되 메일은 안 쏘는, 가벼운 대역이에요. (엄밀히는 아무 일도 안 하니 dummy에 가까운 fake지만, "진짜 자리를 채우는 실동작 구현"이라는 성격상 fake로 보면 이해가 편해요. 이 대역이 *왜* 필요했는지는 〈04〉에서 제대로 풀어요.)

**둘 — `@MockitoBean NotificationMailSender mailSender` (mock).**
```java
@MockitoBean
NotificationMailSender mailSender;                                          // ❸
```
`❸`이 이번 단계의 진짜 주인공이에요. 이건 아무것도 안 채우는 대역이 아니라, **호출을 기록해 두고 나중에 심문당하는** mock이죠. 그래서 테스트 끝에서 이렇게 캐물을 수 있어요.
```java
verify(mailSender).send(eq("alice@example.com"), anyString(), anyString()); // ❹
```
`❹`은 "야, 너 `alice@example.com`으로 `send` 불린 적 있지?"라고 mock을 심문하는 줄이에요. 진짜 발송기로는 절대 못 하던 그 확인이, mock이라서 가능해집니다.

## 왜 fake 하나, mock 하나 — 둘 다 있나

지금은 "둘 다 있다"만 눈에 담아 두세요. 이 둘이 **서로 다른 일**을 하기 때문에 공존한다는 게 이번 [2]의 마지막 그림인데, 그건 재료(〈04〉 fake의 존재 이유, 〈05〉 mock의 메커니즘)를 다 본 뒤 〈06〉에서 맞춰야 선명해져요. 지금 당장은 이 한 줄만 — **"조용히 자리만 채우는 fake"와 "호출을 심문당하는 mock"이 한 프로젝트에 같이 산다.**

그런데 이 대역들을 진짜 발송기 자리에 *어떻게* 끼워 넣는 걸까요? 아무 코드나 마음대로 갈아끼울 순 없잖아요. 그걸 가능케 하는 구조 — **인터페이스·다형성·의존성 주입** — 이 다음 〈03 갈아끼우기를 가능케 한 구조〉의 이야기예요. 사실 이 구조가 없으면 위의 대역들은 애초에 존재할 수조차 없어요.
